## Page Architecture

### Systems Page (`/systems`)

Two visual states driven by `?system=` URL search param:

**State 1: List view** (`?system=` absent)
- Visible `<h1>Systems</h1>` at 20px semibold
- One card row per unique system, sorted descending by automation count
- Each row: `SystemBadge` on the left, system name at 14px regular, count badge on the right ("N automations")
- Entire row is a `<button>` — click calls `setSearchParams({ system: name })`
- Row hover: `hover:bg-secondary/50` — matches AlleAutomatiseringen row hover
- Loading state: `Loader2` spinner centered, `text-muted-foreground`, same as AlleAutomatiseringen

**State 2: Detail/filtered view** (`?system=HubSpot`)
- Back button at top: "← Back to Systems" — plain text button, `text-sm text-primary hover:underline`
- `<h1 className="sr-only">Systems</h1>` (NAV-02 compliance maintained; the back button and system name serve as visible context)
- Selected system name displayed as `<SystemBadge>` + heading: e.g. `SystemBadge` then `<span className="text-xl font-semibold ml-2">HubSpot</span>` — inline flex row
- Automation rows: exact same card pattern as AlleAutomatiseringen (`bg-card border border-border rounded-[var(--radius-outer)] shadow-sm`)
- Each automation row shows: ID (mono xs), name (font-normal truncate), CategorieBadge, StatusBadge, VerificatieBadge, expand chevron
- Expanded row: same `AnimatePresence` + `framer-motion` accordion as AlleAutomatiseringen

### Owners Page (`/owners`)

Mirrors Systems page structure, replacing system-specific elements:

**State 1: List view** (`?owner=` absent)
- Visible `<h1>Owners</h1>` at 20px semibold
- One card row per unique owner (skip empty-string owners), sorted descending by automation count
- Each row: `Users` icon (16px, `text-muted-foreground`) on the left, owner name at 14px regular, count badge on the right
- Entire row is a `<button>` — click calls `setSearchParams({ owner: name })`

**State 2: Detail/filtered view** (`?owner=Jan`)
- Back button: "← Back to Owners"
- `<h1 className="sr-only">Owners</h1>` (NAV-02 compliance)
- Selected owner displayed: `Users` icon + `<span className="text-xl font-semibold ml-2">{name}</span>`
- Same automation card rows as Systems detail view

### Nav Items (AppLayout.tsx "Systems & People" group)

Add to the currently empty `items` array in the "Systems & People" group:

| Title | URL | Icon |
|-------|-----|------|
| Systems | /systems | Server (lucide-react) |
| Owners | /owners | Users (lucide-react) |

Nav item text must match page h1 character-for-character (NAV-02). Active state uses existing sidebar active styling.

Top bar label is resolved automatically: `navGroups.flatMap(g => g.items).find(n => n.url === location.pathname)?.title` will find "Systems" and "Owners" once items are added to navGroups.

## Interaction Contract

### URL State

| Action | URL Result |
|--------|-----------|
| User lands on /systems | `?system=` absent — list view shown |
| User clicks a system row | `?system=HubSpot` set — filtered view shown |
| User clicks "← Back to Systems" | `setSearchParams({})` — list view restored |
| User navigates away and back | List view shown (search param cleared by navigation) |
| Phase 8 deep-link from detail panel | `/systems?system=HubSpot` — filtered view loads directly |

Same pattern applies to /owners with `?owner=` param.

### Accordion (detail views)

Identical to AlleAutomatiseringen:
- `useState<string | null>` for `openId`
- Click toggles: `setOpenId(isOpen ? null : a.id)`
- `AnimatePresence` wraps `motion.div` with `initial={{ height: 0, opacity: 0 }}`, `animate={{ height: "auto", opacity: 1 }}`, `exit={{ height: 0, opacity: 0 }}`
- Spring: `{ type: "spring", stiffness: 300, damping: 30 }`
- Inner padding: `px-4 pb-4 pt-2` (16px horizontal and bottom)

### Loading

- Show `Loader2` spinner while `isLoading === true`
- Spinner is centered: `<div className="flex items-center justify-center py-20">`
- Icon: `<Loader2 className="h-6 w-6 animate-spin text-muted-foreground" />`

## Implementation Notes for Executor

1. **Two new page files:** `src/pages/Systems.tsx` and `src/pages/Owners.tsx`. Both follow the two-state pattern: entity list when no search param, filtered automation list when param is set.

2. **Route registration:** Add `<Route path="/systems" element={<Systems />} />` and `<Route path="/owners" element={<Owners />} />` inside the `ProtectedRoutes` block in `App.tsx`.

3. **Nav items:** Populate the currently empty `items: []` array in the "Systems & People" `navGroups` entry in `AppLayout.tsx`. Import `Server` and `Users` from `lucide-react`.

4. **Derivation guard:** Skip empty-string owners (`if (a.owner?.trim())`). Systems use the `Systeem` TypeScript type so casing is consistent — no normalization needed.

5. **Sort order:** Both entity lists sort descending by count (highest count first). No alphabetical fallback needed.

6. **h1 compliance (NAV-02):** Systems list view: visible `<h1 className="text-xl font-semibold">Systems</h1>`. Systems detail view: `<h1 className="sr-only">Systems</h1>`. Mirrors pattern from Phase 6 (sr-only for full-context pages, visible for clean list pages).

7. **No edit/delete in entity pages:** Automation rows in detail views are read-only accordion cards. Do not render Edit or Delete buttons — those actions live in All Automations only.

8. **SystemBadge reuse:** Use existing `SystemBadge` from `@/components/Badges` for system identifiers in list rows. Do not render raw text.

9. **Top bar auto-resolution:** Once nav items are added to `navGroups`, the top bar `flatMap` lookup will resolve "Systems" and "Owners" automatically. No additional top bar changes needed.

10. **Font weight in rows:** Entity name in list rows uses `font-normal` (400). Do not use `font-medium` — only two weights are in contract: 400 (regular) and 600 (semibold).
